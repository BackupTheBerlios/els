Communication

Before starting submitting patches it would be helpful if you join the developer mailing list and share your ideas. This can save double work.

Two steps

The best option to support patches has two steps. In the first step, you do a CVS checkout.

Then you modify the code. The makefile help you to create test-RPMs. You can try them. Once you killed all obvious errors you should do a 

  cvs diff -u >my-new-feature.patch

and submit this to the patch manager.

Becoming a developer

This project is open --- that means: our list of developers should not be static. After some patches you can become a developer in the project with your own CVS access.
